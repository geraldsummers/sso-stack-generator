# Mission

## The Problem

Small groups need serious coordination tools, but the usual path is to spread
identity, files, docs, passwords, memory, and workflows across corporate SaaS
platforms. That can be practical at first, but it often leaves the group with
weak ownership and no clear way to leave.

## The Pattern

This project demonstrates a replicable pattern: private open-source
infrastructure for small teams, generated from source, deployed as a
secret-free bundle, protected by shared identity, operated through visible
system services, and verified after deployment.

## What Sovereignty Means Here

Sovereignty here does not mean isolation. It means the ability to own critical
operational systems, leave vendors when needed, and keep organizational memory
under local control.

It also means being honest about operations: backups, restores, updates,
offboarding, documentation, and support boundaries matter.

## What This Project Is

This project is a practical infrastructure proof for small teams that want
client-owned systems:

- shared identity
- private docs
- files
- passwords
- observability
- backups
- service catalog
- optional AI/data workspaces
- verification
- restore and update documentation

## What This Project Is Not

This is not a one-click SaaS clone, a generic managed IT service, or a promise
that operations disappear. It is not a formal compliance product by itself. It
is not the right fit for every business.

## Why Open Source Matters

Open source makes the operating model inspectable. A client can see how the
stack is built, where secrets are rendered, how routes are protected, how
services are supervised, and how verification works.

## Why Small Groups Matter

Small groups often do important work without dedicated platform teams. They
still need reliable identity, documentation, collaboration, backups, and
operational memory. The goal is to make small groups powerful without making
them dependent.
